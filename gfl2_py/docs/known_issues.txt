                  GFL2 Parser — Known Issues


1.  Blank scores for rows following a new-buff asset save (Windows)

   STATUS: Persistent / Unresolved

   FIRST OBSERVED: Session processing gm_250818.png (June 2026).
   Rows 2, 3, 4, 8, 10 — every row whose PREVIOUS row triggered a
   buff_ocr._save_asset() call — produced blank score fields in the CSV.

   SYMPTOMS:
   - Score column empty in CSV for a repeatable subset of rows.
   - Affected rows are not random; they always follow a row where a new
     buff PNG was written to assets/buff/.
   - Not reproduced in the Linux sandbox — only on the user's Windows
     machine (Python 3.14, Tesseract from UB-Mannheim).

   CONFIRMATION VIA TRACE (GFL2_TRACE=1, gm_250818.png):
   - Clean rows:   _ocr_score takes ~0.154s (method 1 succeeds).
   - Affected rows: _ocr_score takes ~0.303s (method 1 fails, method 2
     also attempted and also fails).
   - The 0.303s rows correspond exactly to rows immediately following a
     new-buff asset save in the previous row.

   ASSUMPTIONS / DEDUCTIONS:
   - Writing a PNG file to assets/buff/ via cv2.imwrite() on Windows
     triggers filesystem activity (NTFS journal, antivirus scan, or
     page-cache pressure) that degrades Tesseract's ability to write
     or read its own temp files for the next OCR call.
   - The contamination is cross-row: row N's asset save affects row
     N+1's score OCR, not row N's own score (which was already computed).
   - Both OCR methods (image_to_string and image_to_data) fail, ruling
     out a PSM-mode-specific issue.
   - The issue does not reproduce in Linux, pointing to a Windows-specific
     Tesseract / NTFS interaction.

   ATTEMPTS TO RESOLVE:
   a. Move _ocr_score before buff_ocr.translate in _parse_row.
      Result: FAILED. Cross-row contamination means row N's score is
      already computed when row N-1's asset save causes the problem.

   b. Reorder execution so score extracts before buff_ocr in _parse_row.
      Result: FAILED for same reason as (a).

   c. compile_gfl2.py with CHECKED_HASH .pyc invalidation to ensure
      new code is actually loaded on Windows.
      Result: PARTIAL. Helped confirm code changes were taking effect,
      but blank scores persisted.

   d. Batch ALL score and name OCR calls before any buff_ocr calls in
      parse() (current implementation in weekly_gunsmoke.py).
      In theory this completely decouples Tesseract from asset writes.
      Result: UNCONFIRMED. Works in Linux sandbox. Still failing on
      Windows. Possible explanation: stale .pyc on user's machine loading
      the old unbatched code despite compile_gfl2.py being run.

   IDEAS TO GATHER MORE FACTS:
   - Add score value to trace output (log what _ocr_score returns, not
     just its elapsed time) to confirm whether the function returns None
     or an empty string.
   - Add a trace log inside _save_asset to confirm it is actually called
     before the failing _ocr_score, establishing the cross-row timeline.
   - Check whether Tesseract's TESSDATA_PREFIX or temp directory is set
     to a network/NTFS path on the user's machine (network paths are
     known to cause Tesseract hangs).
   - Try setting TMPDIR / TMP / TEMP to a local (non-NTFS) path before
     running main.py to isolate filesystem-specific temp file issues.
   - Print the raw pytesseract return value (before regex filtering) for
     failing rows to distinguish "OCR returned empty" from "OCR returned
     garbage digits that didn't match the score pattern".
   - Try pytesseract with `nice=0` or `timeout=10` parameters to rule
     out a process-level timeout causing silent failures.

   IDEAS TO RESOLVE:
   - Non-Tesseract score extraction: the score is always a 3-6 digit
     number next to a gold coin icon. Use contour detection or template
     matching on digit glyphs to extract it without Tesseract entirely.
   - subprocess isolation: run each Tesseract call in a fresh subprocess
     (`subprocess.run([sys.executable, "-c", "import pytesseract..."])`)
     to guarantee no state is shared between calls.
   - Inter-row sleep: add a time.sleep(0.1) after each buff_ocr call
     that triggered an asset save, to give the filesystem time to settle
     before the next Tesseract invocation.
   - pytesseract.pytesseract.run_tesseract cleanup: explicitly call
     pytesseract's internal cleanup between rows to force temp-file
     handles to close.


2.  Owner name OCR noise (minor)

   STATUS: Accepted / Low priority

   SYMPTOM: Owner name occasionally has 1-2 leading noise characters
   (e.g. "j GreenMan" instead of "GreenMan"). Also sometimes has
   trailing " -" from the Decisive Battle badge text.

   ASSUMPTION: A small rank icon at the left edge of the name cell
   bleeds into the OCR crop region and is misread as a letter.

   CURRENT MITIGATION: _ALNUM_RE filter strips non-alphanumeric
   characters, which removes most noise. The leading "j" or "f" that
   passes the filter is alphabetic and not stripped.

   IDEA TO RESOLVE: Maintain a known-names allowlist. After OCR,
   fuzzy-match the result against known owner names and correct minor
   misreads. Alternatively, increase the name crop's x0 ratio slightly
   (~0.085 instead of 0.073) to skip the rank icon area.


3.  Date OCR fails on certain top date bands

   STATUS: Partially mitigated

   SYMPTOM: The topmost date band (height ~35-40px, smaller than
   typical 64px date separators) occasionally misreads. For example,
   "08/02" was read as "9/09" with PSM 6.

   MITIGATION: Running PSM 11 first (handles sparse thin-band text),
   then PSM 6 fallback, resolves most cases. The _DATE_BAND_MIN_H
   threshold was also lowered from 40 to 30 to catch 35px bands.

   REMAINING GAP: Some top bands with unusual font rendering may still
   misread. No systematic failure observed in the three processed images.

   IDEA: If OCR fails for the top band, fall back to extracting the
   date from the filename (e.g. "gm_250801.png" → "08/01") as a last
   resort.
